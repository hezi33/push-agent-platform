# 网易新闻 Push 数据分析 — 业务流程还原与 AI Agent 机会点调研

---

## 一、业务流程还原

### 1.1 业务背景

网易新闻 APP 每日通过华为、小米、OPPO、VIVO、三星等手机厂商的推送通道，向数千万用户发送新闻通知（Push）。Push 业务是网易新闻获取用户活跃度和留存的核心手段，日均发送量约 **2,600 万条**，展示用户数约 **760 万**，首启用户数约 **15.7 万**，承载着日均 110 万首启用户的 KPI 目标。我所在的团队负责对 Push 全链路数据进行日常监控、异常归因和策略优化建议输出，直接服务对象是新闻组的编辑老师和业务运营团队。

### 1.2 数据监控指标体系

我每天监控的核心指标体系围绕「发送 → 到达 → 展示 → 打开」四个环节构建，涵盖以下关键指标：

**漏斗转化指标：**

| 漏斗环节 | UV 口径指标 | PV 口径指标 |
|---------|------------|------------|
| 发送（Send） | 发送用户数 | 发送次数 |
| 到达（Arrive） | 到达用户数 | 到达次数 |
| 展示（Show） | 展示人数 | 展示次数 |
| 打开（Open） | 打开人数（Open UV） | 打开次数（Open PV） |

**衍生效率指标：**
- UV 打开率 = 打开人数 / 展示人数（全量约 3.77%）
- PV 打开率 = 打开次数 / 展示次数（全量约 0.72%）
- 人均展示次数 = 展示次数 / 展示人数（全量约 9.65 次）
- 人均打开次数 = 打开次数 / 打开人数（约 1.85 次）
- 到达率 = 到达人数 / 发送人数
- 展示率 = 展示人数 / 到达人数

**核心 KPI 指标：**
- **首启用户数**（First Open UV）：当日首次通过 Push 启动 APP 的用户数，日均目标 110 万，其中全量约 2,600、个性化实时约 51,000、个性化非实时约 226,000、本地实时约 59,000
- **首启率** = 首启用户数 / 展示次数
- **DAU**：约 358 万
- **Push 打开 UV 占 DAU 比例**：约 7.8%

**内容运营指标（轮播池）：**
- 曝光量、曝光用户量、曝光点击量（CTR）
- 浏览时长（汇总浏览时长 / 每 DAU 阅读时长）
- 次均浏览时长、次均浏览进度
- 103/104 位置点击次留率

### 1.3 多维度分析框架

一份日报需要从至少六个维度进行拆分对比：

1. **发送类型维度**：全量（全国统一推送）、本地实时（如「北京暴雨预警」）、个性化实时（基于用户画像的千人千面推送）、个性化非实时（精编/非精编内容）、热点追踪
2. **平台维度**：iOS、Android、Android 去除华为（因华为使用自有 HMS 通道，到达率与展示逻辑不同）
3. **厂商维度**：华为、小米、OPPO、VIVO、三星，关注各厂商的到达率差异和打开率差异
4. **省份/区域维度**：广东、浙江、江苏、山东、北京、上海等 30+ 省份，各省份有独立的目标打开 UV（如广东日均目标约 40,000）
5. **行业/频道维度**：娱乐、体育、社会、财经、科技、教育、军事等，不同行业的 Push 打开率差异显著
6. **内容属性维度**：正能量/负能量、重大主题/非重大主题、调性（左/中/右），用于评估内容质量

### 1.4 日常工作流程

**Step 1 — 数据源准备（每天上午 9:00-10:00）**

每天从四个数据平台下载前一天的数据文件：
- **PUSH 平台**：下载 5 个原始数据文件（全量 Push、个性化 Push、本地 Push、省份维度完成情况、本地编辑工作完成情况），每个文件以时间戳命名（如 `全量push(2024-07-16 00_00_00至2024-07-17 00_00_00).xlsx`）
- **数易（数据分析平台）**：下载 4 个分析报表（Push 关键指标波动分析、各推送类型推送概况、各平台推送概况、PUSH-KPI 报表）
- **编辑排班系统**：各小组的分工表（热推组、页面组、增长组、本地 104 组）
- **内容运营平台**：新内容运营日报 V9（包含轮播池明细和汇总数据）

**Step 2 — 文件校验（Python 自动化）**

运行 `push日报-V19.2.0.py` 脚本，自动执行以下校验：
- 检查每个文件的最后修改时间是否为当天，确保数据已更新
- 验证文件名中的日期与目标日期是否一致
- 检测是否存在同名重复文件或缺失文件
- 校验不通过时，脚本会打印具体的异常文件清单并暂停，等待人工检查后确认继续或退出

**Step 3 — 数据处理与指标计算（Python 自动化，约 5 分钟）**

脚本读取所有源数据，自动完成：
- 数据清洗（去除空行、统一日期格式、列重命名）
- 按发送类型（all/全量/个性化实时/本地实时/个性化非实时/other）分别计算各 funnel 指标
- 从 KPI 报表中提取独立打开 UV 和首启 UV 数据
- 按平台维度（iOS/Android/Android 去除华为）拆分指标
- 按厂商维度（华为/小米/OPPO/VIVO/三星）拆分指标
- 处理替换 Push 和补发 Push 的特殊逻辑（同一文章可能被多次推送覆盖，需合并统计）
- 计算每个编辑的内容贡献度（贡献度 = 总打开量² / 总展示量）
- 将当日数据追加到历史日报 `push日报.xlsx` 的各 Sheet 中，形成时间序列
- 生成最终多 Sheet Excel 输出（约 15+ Sheet）

**Step 4 — 可视化报表生成（约 10 分钟）**

从 Excel 报表中提取关键数据，用 Python 图表库（如 pyecharts/matplotlib）生成 HTML 可视化报告，包含：
- 各核心指标的逐日折线图（近 7/14/30 天趋势）
- 环比/同比变化柱状图（如 UV 打开率环比昨日 -0.5%）
- 各发送类型的指标对比雷达图
- 各厂商/省份的排行榜柱状图
- 漏斗转化率桑基图（发送→到达→展示→打开）

**Step 5 — 异常分析与归因（人工，30-90 分钟）**

这是整个工作流中最耗时的环节。发现数据异常后（如某天 UV 打开率突然下降 15%），我需要进行以下归因分析：

- **第一步 — 确认异常范围**：是整体下降还是局部下降？先在总览表确认，再按发送类型、平台、厂商、省份逐一排查，定位到「个性化实时 × Android 去除华为 × 小米 × 广东省」这种粒度
- **第二步 — 链路定位**：在漏斗的四个环节中确定问题节点。例如：到达率没变但展示率下降 → 厂商通道的展示策略有调整；展示量没变但打开率下降 → 内容质量问题。我会在 SQL 中查询各节点的转化率时序数据（`arrive_rate`, `show_rate`, `open_rate`）
- **第三步 — 下钻分析**：如果是内容质量问题，继续查看该维度下具体发了哪些文章，标题、摘要、分类、发送量、打开率如何。如果是厂商问题，对比同期的其他厂商数据排除全局因素的影响
- **第四步 — 外部因素排查**：检查是否有竞品大事件、节假日效应、APP 版本更新、厂商推送策略调整等外部干扰

**Step 6 — 撰写分析报告与交付（约 30 分钟）**

将分析结论整理成报告，包含：
- 昨日核心数据概览（与前一天/上周同期对比）
- 异常指标标注及变化幅度
- 归因分析结论（定位到具体维度和原因）
- 优化建议（如「建议减少某类低打开率内容的全量推送」「某省份本地编辑条数不足需补充」）
- 将 Excel 报表和 HTML 图表一并发送给编辑组老师

### 1.5 辅助工作：SQL 按需取数

除了日报流程外，业务方经常提出临时数据需求，如：
- 「查一下五月份小米用户的 open UV，按发送类型拆分」
- 「对比一下华为和 OPPO 用户在过去两周的人均打开次数变化」
- 「拉一下广东省上周本地实时 Push 的标题和对应的打开率」

这类需求需要我手动编写 SQL，从内部数据库（如数易系统）中提取数据。数据库表结构包含推送明细表（push_send_detail，含 pushId、发送时间、发送类型、厂商、平台、省份、发送量、到达量、展示量、打开量等字段）、推送内容表（标题、摘要、分类、频道等）、用户行为表（DAU、启动次数、浏览时长等）。我写好 SQL 后在查询平台执行，导出结果再用 Python 做简单的聚合和格式化，最终返回给需求方。

## 二、痛点分析

### 痛点 1：异常发现的滞后性——出问题2小时后才知道

**场景描述：** 某天上午 11 点，本地实时 Push 的到达率突然从正常的 35% 骤降至 22%，但我直到第二天早上跑日报时才发现。原因是厂商通道在前一天做了策略调整，导致近 10 个小时的低效推送无人感知。

**具体问题：** 当前所有数据监控依赖 T+1（次日）报表，没有实时告警机制。所有异常都是事后发现的，缺乏分钟级或小时级的主动监控能力。

**影响：** 异常持续时间越长，损失的用户量和首启量越大。以本地实时日均 5.9 万首启计算，到达率下降 13 个百分点可能意味着当天损失约 7,600 个首启用户。

### 痛点 2：归因分析靠手工——排查一个异常平均耗时 1-2 小时

**场景描述：** 某天全量 Push 的 UV 打开率环比下降 0.5 个百分点，我需要依次排查：是不是某个厂商下降了？是不是某个省份下降了？是不是某些文章打开率特别低？是不是到达环节出了问题？每次都需要手动在 Excel 中做数据透视、对比、下钻。

**具体问题：** 归因没有自动化路径，完全依赖分析师的个人经验和直觉。需要手动逐个维度切分、逐个环节排查，链路长、维度多，穷举所有组合几乎不可能（6 种发送类型 × 5 个厂商 × 30+ 省份 × 4 个漏斗环节 = 3600+ 种组合）。

**影响：** 每天花在归因上的时间占工作时长的 50% 以上，压缩了真正有价值的策略建议输出时间。且由于时间限制，很多细颗粒度的异常根本来不及排查。

### 痛点 3：维度拆分繁琐重复——每天做 80% 一样的事

**场景描述：** 每天早上打开电脑 → 下载文件 → 运行脚本 → 看总览 → 看全量 → 看个性化 → 看本地 → 看 iOS → 看 Android → 看华为 → 看小米 → 看 OPPO → … 每天在同样的 Excel Sheet 之间来回切换，做着同样的对比操作。

**具体问题：** 日报流程的标准化程度很高（Python 脚本已覆盖 70%），但数据解读和维度对比仍需人工逐一查看。SQL 取数需求也高度重复——80% 的临时需求只是换了时间和维度，本质上查询逻辑相同。

**影响：** 重复劳动严重消耗精力，一个实习生的日常有 60% 的时间在做「人肉 SQL 翻译机」和「人肉报表生成器」，数据分析师本应做的高价值洞察工作反而被挤占了。

### 痛点 4：SQL 取数需求响应慢——业务方要等半天到一天

**场景描述：** 编辑老师下午 3 点发来消息：「帮我查一下上个月广东省华为用户各发送类型的打开 UV，2 点开会要用。」我放下手头工作去写 SQL，查询跑完还要导出、清洗、格式化，一套流程下来至少 30 分钟。如果需求复杂涉及多表关联，可能 1-2 小时才能交付。

**具体问题：** 编辑老师不懂 SQL，我只能充当「人肉查询接口」。所有取数需求都要经过我，形成了单点瓶颈。而且 SQL 逻辑每次都要重新编写，缺乏复用机制。

**影响：** 业务方取数等待时间长，决策效率低。取数需求的堆积也让我无法专注于分析工作，陷入「一直在跑 SQL，但没时间思考数据」的困境。

### 痛点 5：分析结论难以推动落地——建议给了但不知道有没有执行

**场景描述：** 我在报告中指出「某省本地 Push 条数连续三天低于目标值 30%，建议编辑增加该省本地新闻的推送频次」，但一周后复查数据时发现情况没有改善。我不知道是编辑没看到建议、看到了但没时间处理、还是压根不认可这个建议。

**具体问题：** 从数据发现到策略执行之间缺少闭环机制。分析报告发出后，没有跟踪建议是否被采纳、执行效果如何。分析师和编辑之间是单向的信息传递，缺乏反馈回路。

**影响：** 分析工作的价值无法被量化和追踪。很多有价值的洞察停留在报告中，没有转化为实际的业务改进。长此以往，数据分析的价值感减弱。

### 痛点 6：归因深度受限——只能定位到「什么变了」，很难解释「为什么变」

**场景描述：** 个性化实时 Push 的人均展示次数在某天突然从 5.1 次下降到 4.3 次。我能查到是「小米」和「OPPO」两个厂商的展示量下降了，但为什么下降——是厂商推送策略变了？是用户主动关闭了通知权限？还是特定机型的 SDK 出了问题？这些我无法从现有数据中回答。

**具体问题：** 现有的数据体系只能呈现「现象层」的指标变化，缺乏对根因的深度挖掘能力。需要联系厂商侧的技术同学、查看 SDK 上报日志、甚至等厂商反馈才能确认原因，跨团队沟通成本高、周期长。

**影响：** 分析报告的结论往往是「XX 指标下降，原因为 XX 维度表现不佳」，但无法给出可操作的修复建议。业务方听到「厂商推送策略变化」这种结论时，下一步行动依然是——等待和观察。

## 三、AI Agent 机会点

### 设计思路：为什么是四个 Agent

在设计 Agent 之前，先对六个痛点做一次分类——按「受众是谁」和「触发方式」两个维度来看：

| 痛点 | 核心受众 | 触发方式 | 对应的 Agent 能力 |
|------|---------|---------|-----------------|
| 痛点 1：异常发现滞后 | 分析师 | 主动（系统自动巡检） | 监控 |
| 痛点 2：归因靠手工 | 分析师 | 被动（发现异常后手动排查） | 归因 |
| 痛点 6：归因深度受限 | 分析师 | 被动 | 归因 |
| 痛点 3：重复劳动多 | 分析师 | 被动（每天固定流程） | 监控 + 归因（解放人力） |
| 痛点 4：SQL 取数慢 | **编辑老师** | **被动（编辑提问触发）** | **问答（独立通道）** |
| 痛点 5：结论难落地 | 分析师 + 编辑 | 被动（报告发出后无跟进） | 策略 |

关键发现：**痛点 4（取数慢）的受众、触发方式、技术路径都和其他痛点完全不同**——其他痛点走的是「分析师发现异常→排查→建议」的分析链路，而取数需求是编辑老师直接发起的、高频的、与异常分析无关的诉求。因此将「自然语言取数」独立为一个 Agent，而不是塞进策略 Agent 的附属功能。

最终形成 **四个 Agent、两条通道**：

```
分析通道（主动链路）：
  ① 监控 Agent → ② 归因 Agent → ④ 策略 Agent

自助通道（被动链路）：
  ③ 问答 Agent（编辑直接使用，不经过分析师）
```

---

### 3.1 监控 Agent — 从"事后发现"到"实时预警"

**对应痛点：** 痛点 1（异常发现滞后）、痛点 3（重复劳动）

**一句话职责：** 自动巡检数据，发现异常主动告警。

**Agent 能力设计：**

监控 Agent 以小时甚至分钟级频率自动巡检各维度核心指标，建立基线模型（基于历史同期数据和周期性规律），一旦检测到指标偏离正常波动范围（如 UV 打开率较 7 天同期均值下降超过 2 个标准差），主动向分析师和编辑推送告警：

> 🚨 **[PUSH 异常告警] 2024-07-17 14:00**
> 本地实时 Push 到达率骤降至 22%（前 7 天同期均值 35%，下降 37%）
> 影响范围：Android-小米 通道
> 影响省份：广东（-15%）、浙江（-12%）
> 预计已损失首启用户约 2,300
> 建议：已自动触发归因 Agent，分析报告预计 5 分钟后生成

**人工方式 vs Agent 方式对比：**

| 维度 | 人工方式 | Agent 方式 |
|------|---------|-----------|
| 发现速度 | T+1（次日） | 小时级/分钟级 |
| 覆盖范围 | 只能盯核心指标（5-10 个） | 可同时监控数百个维度组合 |
| 准确性 | 依赖个人经验判断 | 基于统计模型的客观异常判定 |
| 值班成本 | 需人时刻盯着 | 7×24 自动运行 |
| 告警后动作 | 手动开始排查 | 自动触发归因 Agent |

**核心价值：** 将异常发现时间从「次日」提前到「实时」，在问题发生的前 30 分钟内就触发响应，大幅缩短异常持续时间，减少用户和首启损失。

---

### 3.2 归因 Agent — 从"手工排查 2 小时"到"自动定位 5 分钟"

**对应痛点：** 痛点 2（归因耗时）、痛点 6（归因深度受限）

**一句话职责：** 自动切维度、查关联、定位根因，输出结构化归因报告。

**Agent 能力设计：**

归因 Agent 在接收到监控告警（或分析师手动触发）后，自动执行多层级归因分析：

1. **范围锁定**：自动在所有维度上做贡献度分解（类似决策树），快速定位异常集中在哪个维度组合。例如：全量 UV 打开率下降 0.5pp → 其中 80% 的贡献来自 Android 小米（-1.2pp），而 iOS 反而微增（+0.1pp）→ 小米中广东省贡献了 60% 的降幅 → 广东省昨天下午 17:00-19:00 时段降幅最大。

2. **链路定位**：在锁定维度后，自动检查漏斗四个环节的转化率变化，判断是到达问题（厂商通道）、展示问题（SDK/客户端）还是打开问题（内容质量/用户兴趣）。

3. **根因推断**：结合上下文信息（该维度昨天发了什么文章、文章标题的 CTR 预测、竞品 Push 情况、该厂商近期通道稳定性数据、该省份是否有特殊事件等），给出可能的根因假设和置信度。

4. **生成归因报告**：自动输出一份结构化的归因分析，包含异常定位、根因假设、支撑数据、置信度评分。报告可直接插入日报或作为独立文档发送。

**人工方式 vs Agent 方式对比：**

| 维度 | 人工方式 | Agent 方式 |
|------|---------|-----------|
| 归因耗时 | 1-2 小时 | 3-5 分钟 |
| 覆盖维度 | 手动抽查，无法穷举 | 自动遍历所有维度组合（3600+） |
| 归因深度 | 经验驱动，易遗漏 | 系统化 + 统计方法驱动 |
| 可复现性 | 依赖个人，难以复制 | 标准化流程，每次都一致 |
| 知识沉淀 | 经验在脑子里，人走经验走 | 归因路径可追溯、可复用 |

**核心价值：** 将分析师从重复的「切维度-看数据-比差异」中解放出来，把归因耗时从小时级压缩到分钟级，让分析师能把精力集中在策略建议和业务沟通上。

### 3.3 问答 Agent — 从"人肉 SQL 翻译机"到"编辑自助取数"

**对应痛点：** 痛点 4（SQL 取数响应慢）、痛点 3（重复劳动）

**一句话职责：** 让编辑老师用自然语言直接取数，30 秒拿到结果，不再经过分析师。

**为什么必须从策略 Agent 中独立出来：**

旧版设计中自然语言取数被放在策略 Agent 里作为一个附属功能，但这是不合理的：

1. **受众不同**：问答 Agent 的使用者是**编辑老师**（每天催取数的人），策略 Agent 的使用者是**分析师**和**管理层**。两类用户的需求场景完全不同，不应混在同一个入口。
2. **触发方式不同**：问答是**被动触发**（编辑问了才动），监控→归因→策略是**主动触发**（数据异常自动启动链路）。两条链路的节奏完全不同。
3. **技术路径不同**：问答 Agent 的核心是 NL2SQL + 数据库查询 + 结果格式化，策略 Agent 的核心是规则引擎 + 知识库匹配 + 跟踪系统，两套技术栈没有复用关系。
4. **痛点权重不同**：取数是你每天被催得最烦的事（高频，每天多次），把这个最高频的痛点藏在一个 Agent 的角落功能里，完全低估了它的重要性和独立价值。

**Agent 能力设计：**

编辑老师在聊天窗口直接用自然语言提问，Agent 自动完成翻译→查询→格式化→返回：

> **编辑：** 广东省上个月小米用户的 open UV 是多少？按发送类型拆分。
>
> **问答 Agent：** 📊 2024年6月 | 广东省 | 小米用户
>
> | 发送类型 | Open UV | 环比5月 |
> |---------|---------|---------|
> | 全量 | 12,340 | +3.2% |
> | 本地实时 | 8,920 | -1.5% |
> | 个性化实时 | 45,600 | +8.7% |
> | 个性化非实时 | 23,100 | -0.3% |

整个交互过程 30 秒内完成，编辑不需要理解 SQL、不需要知道表名，也不需要等分析师有空。

**进阶能力：**

- **模糊语义理解**：编辑问「小米用户」可能指「Android 小米厂商用户」，Agent 自动消歧；如果歧义过大（如「打开率」没说是 UV 还是 PV 口径），主动追问澄清
- **上下文记忆**：编辑追问「那华为呢？」Agent 自动继承上一个查询的时间范围、省份、指标，只替换厂商维度
- **例行订阅**：编辑可以订阅「每天早上 9 点把昨天广东省各发送类型的 open UV 发给我」，Agent 定时执行并推送
- **异常标注**：如果查询结果中某个指标偏离正常范围，Agent 在返回结果时主动标注

**人工方式 vs Agent 方式对比：**

| 维度 | 人工方式 | 问答 Agent |
|------|---------|-----------|
| 响应速度 | 30 分钟 ~ 2 小时 | 30 秒 |
| 可用时段 | 分析师上班时间 | 7×24 |
| 瓶颈 | 单点（分析师是唯一通道） | 无瓶颈，编辑直接问 |
| 编辑体验 | 催人 → 等待 → 再催 | 直接问 → 立刻拿结果 |
| SQL 复用 | 每次重新写 | 查询逻辑自动沉淀和复用 |
| 分析师精力 | 被取数打断，无法专注分析 | 解放出来做高价值工作 |

**核心价值：** 解决最高频、最痛的日常问题。问答 Agent 独立成军后，分析师不再是「人肉 SQL 接口」，编辑取数不再有等待成本。这是四个 Agent 中 ROI 最直观、见效最快的一个。

### 3.4 策略 Agent — 从"报告发出就结束"到"建议→执行→验证闭环"

**对应痛点：** 痛点 5（结论难落地）

**一句话职责：** 基于归因结果生成可执行的优化建议，跟踪执行效果，形成数据驱动决策闭环。

**Agent 能力设计：**

策略 Agent 在归因 Agent 输出结论后，完成从「知道问题」到「解决问题」的最后一棒：

1. **建议生成**：根据归因结论自动匹配优化策略。例如——归因结论是「广东省本地 Push 条数连续 3 天低于目标值 30%，导致打开 UV 不达标」→ 策略 Agent 输出：

   > 💡 **优化建议 #2024-0717-01**
   > **问题：** 广东省本地 Push 日均 12 条（目标 20 条），打开 UV 缺口约 1,400/天
   > **建议：** 向广东省增发 5-8 条本地 Push
   > **参考方向：** 近期广东省打开率最高的三类内容——天气预警（2.1%）、交通管制（1.9%）、本地民生新闻（1.7%）
   > **效果预估：** 按平均打开率 1.8% 估算，增加 8 条推送预计提升打开 UV 约 1,200，达成率从 82% 提升至 96%
   > **跟踪周期：** 执行后 3 天
   > **负责编辑：** @张三（广东省早班编辑）

2. **效果预估**：基于历史数据模拟建议执行后的量化效果，帮助编辑和管理层评估优先级。预估不是「拍脑袋」，而是基于同类建议的历史执行效果数据做参照。

3. **执行跟踪**：建议推送给编辑后，Agent 持续监控关联指标：
   - 第 1 天：广东省本地 Push 条数是否增加？（执行检查）
   - 第 3 天：广东省打开 UV 是否改善？（效果检查）
   - 如果 3 天后指标未改善 → 自动触发二次提醒并抄送编辑组长
   - 如果改善了 → 记录策略效果，标注为「有效策略」，用于后续类似场景的推荐

4. **策略效果沉淀**：长期运行后形成策略知识库——哪些建议被采纳了、采纳后效果如何、哪些类型的问题适合哪些策略——让策略推荐越来越精准。

**人工方式 vs Agent 方式对比：**

| 维度 | 人工方式 | Agent 方式 |
|------|---------|-----------|
| 建议质量 | 依赖分析师个人经验 | 基于历史策略效果数据，越用越准 |
| 建议时效 | 分析完后 30-60 分钟 | 归因完成即时生成 |
| 落地跟踪 | 无（报告发出即结束） | 自动跟踪，超时升级 |
| 闭环验证 | 无 | 自动对比执行前后的指标变化 |
| 策略沉淀 | 经验在人脑里，不可复用 | 策略知识库，可持续积累 |

**核心价值：** 补齐从分析到执行的「最后一公里」。策略 Agent 的价值不只是「告诉你怎么办」，而是确保「有人办、办了没、效果好不好」，让数据分析真正驱动业务改进。

### 3.5 四 Agent 协作全景图

核心设计理念：**两条通道，互不干扰**。

```
┌─────────────────────────────────────────────────────────────────┐
│                          数据源层                                │
│   PUSH平台  │  数易平台  │  排班系统  │  内容运营平台  │  内部数据库  │
└────────────┬──────────────────────────────┬──────────────────────┘
             │                              │
             │  主动分析通道                   │  自助取数通道
             │  (异常发现→归因→改进)            │  (编辑直接提问取数)
             ▼                              │
┌────────────────────────┐                  │
│   ① 监控 Agent          │                  │
│   分钟级巡检              │                  │
│   异常检测 → 主动告警     │                  │
│                         │                  │
│   "到达率骤降37%，        │                  │
│    影响小米-广东通道"     │                  │
└────────┬───────────────┘                  │
         │ 告警触发                          │
         ▼                                  │
┌────────────────────────┐                  │
│   ② 归因 Agent          │                  │
│   自动切维度              │                  │
│   漏斗定位 → 根因推断     │                  │
│   输出归因报告            │                  │
│                         │                  │
│   "80%降幅来自广东省       │                  │
│    小米通道到达率下降"     │                  │
└────────┬───────────────┘                  │
         │ 输出归因结论                       │
         ▼                                  │
┌────────────────────────┐                  │
│   ④ 策略 Agent          │                  │
│   建议生成 → 效果预估     │                  │
│   执行跟踪 → 闭环验证     │                  │
│                         │                  │
│   "增发广东本地Push 5-8条 │                  │
│    预计提升打开UV 1,200"  │                  │
└────────────────────────┘                  │
                                             │
         ┌───────────────────────────────────┘
         │  编辑提问触发
         ▼
┌────────────────────────────────────────────┐
│   ③ 问答 Agent                              │
│   自然语言 → SQL → 数据库查询 → 格式化返回    │
│                                             │
│   编辑："广东省上月小米open UV？"              │
│   Agent：📊 [表格即时返回] + 订阅 + 异常标注   │
│                                             │
│   独立通道，7×24可用，不经过分析师              │
└────────────────────────────────────────────┘
```

**通道一 — 主动分析通道（①→②→④）：**
- 触发方式：系统自动巡检或分析师手动触发
- 数据流向：监控发现异常 → 归因定位根因 → 策略推动改进
- 服务对象：分析师和管理层
- 价值：将「发现异常→排查原因→输出建议→跟踪落地」全链路自动化

**通道二 — 自助取数通道（③）：**
- 触发方式：编辑老师自然语言提问
- 数据流向：用户提问 → NL2SQL → 数据库查询 → 即时返回
- 服务对象：编辑老师和所有业务方
- 价值：消灭「人肉 SQL 翻译机」，取数需求不再经过分析师

两条通道完全解耦：编辑问数据不会打断分析链路，分析链路有发现时也不会干扰编辑的正常取数。

### 3.6 落地优先级建议

从「解决最痛的日常问题」出发，优先级排序为：

| 优先级 | Agent | 解决什么 | 为什么排这个位置 |
|--------|-------|---------|----------------|
| 🥇 P0 | **问答 Agent** | 痛点 4：取数慢 | 最高频（每天被催 N 次）、ROI 最直观（30 秒 vs 2 小时）、技术最成熟（NL2SQL 方案完善）、受众最广（所有编辑都用得上）。**先解放分析师，让他有时间干别的。** |
| 🥈 P1 | **归因 Agent** | 痛点 2+6：归因耗时 | 最大时间消耗（每天 1-2 小时）。问答 Agent 上线后分析师腾出的时间，正好用来训练和调优归因 Agent 的分析逻辑。 |
| 🥉 P2 | **监控 Agent** | 痛点 1+3：发现滞后 | 需要归因 Agent 成型后才能承接告警后的分析动作——否则告警一堆但没人排查，告警疲劳会很快出现。 |
| 4️⃣ P3 | **策略 Agent** | 痛点 5：难落地 | 需要前三者积累足够的历史归因数据和策略效果数据，才能做精准的建议匹配和效果预估，适合长期迭代。 |

这个顺序的关键逻辑是：**先把分析师从最机械的劳动中解放出来（问答 Agent），然后让他能高效完成核心分析工作（归因 Agent），再补齐事前预警（监控 Agent），最后打通决策闭环（策略 Agent）。**
