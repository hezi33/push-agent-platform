# 网易新闻 Push 数据分析 Agent 平台

## 项目概览

为网易新闻 Push 推送业务构建的 **AI Agent 数据分析平台**。基于真实业务场景（日均 2,600 万条推送、760 万展示用户、15.7 万首启），通过三个 Agent（监控→归因→策略）实现从「异常发现 → 根因定位 → 策略建议 → 执行闭环」的全链路智能化。

**用户：** 吴丽芳（前网易新闻数据分析实习生，负责 Push 日报）

**当前阶段：** 🎨 原型实现阶段 — 基于已完成的 PRD + 设计规范 + 技术栈文档，开始构建可交互前端原型。

## 业务核心知识

> 以下是从实际数据和调研文档中提炼的关键业务事实，原型设计和开发时需要始终参照。

### 核心指标（基于 2024 年 7 月真实数据）

| 指标 | 数值 | 口径 |
|------|------|------|
| 日均发送量 | 2,600 万条 | 每日所有 Push 发送用户数 |
| 展示人数 | 760 万 | 每日看到 Push 的 UV |
| UV 打开率 | 3.77% | 打开人数 / 展示人数（全量） |
| PV 打开率 | 0.72% | 打开次数 / 展示次数 |
| 首启用户 | 15.7 万/日 | KPI 目标 110 万/日 |
| 人均展示次数 | 9.65 次 | 展示次数 / 展示人数 |
| 人均打开次数 | 1.85 次 | 打开次数 / 打开人数 |
| DAU | 358 万 | 日活跃用户 |
| Push 打开 UV 占 DAU | 7.8% | Push 对 DAU 的贡献 |

### 漏斗链路（四个环节）

```
发送(Send) → 到达(Arrive) → 展示(Show) → 打开(Open)
   └─ 厂商通道 ─┘  └─ SDK/客户端 ─┘  └─ 内容质量/用户兴趣 ─┘
```

**故障定位规则：**
- 到达率 ↓ → 厂商通道问题（联系厂商或调整策略）
- 展示率 ↓ → SDK/客户端问题（技术排查）
- 打开率 ↓ → 内容质量问题（标题/选材/用户画像匹配）

### 六维度分析框架

所有数据分析和异常定位都在这六个维度上展开：

| 维度 | 层级 | 示例值 |
|------|------|--------|
| 发送类型 | 5 类 | 全量 / 本地实时 / 个性化实时 / 个性化非实时 / 热点追踪 |
| 平台 | 3 类 | iOS / Android / Android 去华为 |
| 厂商 | 5 家 | 华为 / 小米 / OPPO / VIVO / 三星 |
| 省份 | 30+ | 广东 / 浙江 / 江苏 / 山东 / 北京 / 上海 / ... |
| 行业频道 | 10+ | 娱乐 / 体育 / 社会 / 财经 / 科技 / 教育 / 军事 / ... |
| 内容属性 | 3 类 | 正能量/负能量 / 重大主题/非重大 / 调性(左/中/右) |

**维度组合数：** 6 × 5 × 30+ × 4(漏斗) ≈ 3600+ 种，这是归因必须自动化的根本原因。

### 六大痛点（设计驱动因素）

| # | 痛点 | 严重程度 | 对应的 Agent |
|---|------|:---:|------|
| 1 | 异常发现滞后（T+1 才发现） | 🔴 | 监控 Agent (P2) |
| 2 | 归因手工排查 1-2 小时（3600+ 维度组合） | 🔴 | 归因 Agent (P1) |
| 3 | 每日 60% 时间做重复劳动 | 🟠 | 监控 + 归因 |
| 4 | SQL 取数慢，编辑要等半小时到半天（最高频） | 🔴 | 问答 Agent (P0，本阶段暂不实现) |
| 5 | 分析建议缺乏执行跟踪闭环 | 🟠 | 策略 Agent (P3) |
| 6 | 归因只能定位"什么变了"，解释不了"为什么变" | 🟠 | 归因 Agent (P1) |

## 原型页面清单

### 五个页面

| # | 页面 | 路由 | 核心功能 | 对应的 Agent 阶段 |
|---|------|------|---------|-----------------|
| 1 | **数据看板首页** | `/dashboard` | 6 个 KPI 数据卡片 + 趋势图 + 漏斗图 + 异常告警列表 | 监控 (S01-S06) |
| 2 | **异常详情页** | `/anomaly/:alertId` | 告警摘要 + 偏离度趋势 + 多维度对比 + 事件时间线 | 监控→归因 过渡 |
| 3 | **归因分析页** | `/attribution/:reportId` | 贡献度决策树 + 漏斗定位 + 根因推断 + 置信度仪表盘 | 归因 (S07-S09) |
| 4 | **策略建议页** | `/strategy/:suggestionId` | 建议卡片 + 效果预估 + D+1/D+3/D+7 跟踪时间线 + 升级流程 | 策略 (S10-S18) |
| 5 | **处理记录页** | `/history` | 历史告警表格 + 多条件筛选 + 统计汇总 + 行展开详情 | 全流程归档 |

### 页面流转关系

```
数据看板 ──点击告警行──→ 异常详情 ──触发归因──→ 归因分析 ──生成策略──→ 策略建议
    ↑                       ↑                    ↑               │
    └───────────────────────┴────────────────────┴─── 处理记录 ←──┘
```

### 18 种业务状态（S01-S18）

原型中所有 Tag、Badge、进度条都需要覆盖以下状态：

| 阶段 | 状态码 | 含义 |
|------|--------|------|
| 监控 | S01-S06 | 巡检中 / 正常 / 关注 / 告警 / 严重告警 / 数据延迟 |
| 归因 | S07-S09 | 归因中 / 归因完成 / 归因待人工 |
| 策略 | S10-S18 | 建议生成中 / 待审核 / 已推送待执行 / 执行中 / 持续跟踪中 / 已闭环 / 超期未执行 / 效果未达标 / 升级处理中 |

## 实现规划

### 阶段 0：纯前端原型（当前阶段 🎯）

**目标：** 不依赖后端，纯前端 + Mock 数据跑通 5 个页面和所有交互流程。

**工作内容：**

1. **项目初始化** — `npm create vite@latest` + Ant Design + ECharts + React Router + Zustand + TanStack Query
2. **主题配置** — 按 [DESIGN.md §二](DESIGN.md) 映射 Ant Design Token
3. **页面开发**（按顺序）：
   - 数据看板首页（包含 ProCard 指标卡片 + 告警列表 ProTable）
   - 异常详情页（ECharts 偏离度折线图 + 多维度柱状图 + Timeline）
   - 归因分析页（决策树 + 漏斗对比 + 置信度 Gauge）
   - 策略建议页（建议卡片 + 跟踪时间线 + 状态流转）
   - 处理记录页（ProTable 高级筛选 + 行展开）
4. **Mock 数据** — 用 MSW 覆盖所有 API，至少准备 3 套数据：正常态 / 单个告警 / 多个告警
5. **交互联调** — 确保页面间跳转、状态流转、轮询/刷新都正常工作

**Mock 数据场景清单：**

| 场景 | 用途 |
|------|------|
| 正常 — 所有指标正常波动 | 默认首页 |
| 关注 — 1 个指标偏离 1.5-2σ | 测试关注级别渲染 |
| 告警 — 到达率骤降 37%（小米·广东） | 主测试场景 |
| 严重 — 多指标同时 ≥ 3σ 偏离 | 严重告警测试 |
| 归因中 — 归因 Agent 运行中 | 进度轮询测试 |
| 归因完成 — 置信度 65% + 候选假设 | 完整归因报告 |
| 策略跟踪 — D+1/D+3/D+7 各阶段数据 | 策略时间线测试 |
| 升级场景 — 逾期未执行 + 效果不达标 | 异常流程测试 |

### 阶段 1：后端接入

FastAPI + MySQL + Redis，对接真实数据源。

### 阶段 2：AI Agent 接入

LangChain + LangGraph + Claude API，实现三个 Agent 核心逻辑。

### 阶段 3：生产化

Docker + CI/CD + 飞书集成。

## 继续开发指南

当在新会话中继续开发时，按以下步骤快速进入状态：

### Step 1 — 建立全局认知

1. 读本文件（CLAUDE.md）了解项目概览和当前阶段
2. 读 [_agent_prd_push.md](_agent_prd_push.md) 了解三个 Agent 的职责边界、工作流和状态机
3. 读 [DESIGN.md](DESIGN.md) 了解页面设计和交互规范
4. 读 [TECH.md](TECH.md) 了解技术选型和项目结构

### Step 2 — 确认当前进度

- 如果 `frontend/` 目录不存在 → 从阶段 0 开始（项目初始化）
- 如果 `frontend/` 存在但页面不完整 → 继续开发缺失页面
- 如果 5 个页面都完成 → 进入阶段 1（后端）或优化阶段 0

### Step 3 — 开发时的关键约束

- **色彩**：严格使用 [DESIGN.md §二](DESIGN.md) 的 Token，禁止硬编码色值
- **状态**：所有 Agent 状态使用 S01-S18 状态码，Tag 颜色遵循语义色规则
- **置信度**：任何 AI 输出都必须显示置信度（0-100），< 60% 红色 + 「建议人工复核」
- **数据**：原型阶段所有数据来自 MSW Mock，API 端点参见 [TECH.md §3.2](TECH.md)
- **图表**：统一用 ECharts，echarts-for-react 封装，样式遵循数据可视化色板

### Step 4 — 提交规范

- 每个页面开发完成后，确保该页面在 3 种 Mock 场景下都能正常渲染
- 页面间跳转链路完整可走通
